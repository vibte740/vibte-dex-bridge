1. Overview

The Vibte DEX Bridge is a TypeScript-based automated trading system designed to execute TradingView Supertrend v6 signals on decentralized exchanges (DEXs). The system acts as a middleware between TradingView webhooks and DEX APIs, translating trading signals into executable orders on Hyperliquid and dYdX v4.

Built with modern web technologies, the bridge prioritizes security, reliability, and extensibility. The architecture follows a modular, layered approach that separates concerns between webhook handling, validation, risk management, and exchange execution.

2. Network Topology

High-Level System Architecture

The system operates as a standalone service that communicates with three primary external entities:

  • TradingView: Sends HTTP POST requests containing trading signals to the bridge's webhook endpoint
  • DEX APIs: Hyperliquid and dYdX v4 REST/WebSocket endpoints for market data and order execution
  • External Services: Optional Redis for rate limiting, logging systems

Internal Component Topology

Internally, the bridge follows a request-processing pipeline topology:

  1. Entry Point (index.ts) initializes the Express server and sets up graceful shutdown handling
  2. Express Application (server.ts) configures middleware and routes
  3. Webhook Route (routes/webhook.ts) receives and validates incoming TradingView signals
  4. Middleware Layer processes requests through validation and rate limiting
  5. Service Layer orchestrates signal validation, position sizing, and order execution
  6. DEX Client Layer interacts with exchange-specific APIs
  7. Utility Layer provides cross-cutting concerns (configuration, logging)

3. Component Anatomy

The system is organized into distinct layers, each with a specific responsibility:

Layered Architecture

  • Presentation Layer: Express server and routing (server.ts, routes/)
  • Middleware Layer: Request preprocessing (middleware/)
  • Service Layer: Business logic orchestration (services/)
  • Integration Layer: DEX-specific implementations (dex/)
  • Utility Layer: Cross-cutting concerns (utils/)

Core Components

Entry Point (src/index.ts)

Responsible for application bootstrap and lifecycle management:

  • Creates Express application instance
  • Loads configuration and logger
  • Starts HTTP server on configured port
  • Implements graceful shutdown handling for SIGTERM, SIGINT, and uncaught exceptions

Express Application Factory (src/server.ts)

Configures the Express middleware stack and routes:

  • Sets up JSON body parsing
  • Registers global middleware (rate limiting, validation)
  • Defines API routes (/api/webhook)
  • Configures error handling middleware

Webhook Handler (src/routes/webhook.ts)

Processes incoming TradingView signals:

  • Validates webhook secret for authenticity
  • Parses and validates TradingView payload structure
  • Delegates validated signals to the order service
  • Returns appropriate HTTP responses (200 for success, 400/401/429 for errors)

Middleware Components

Request Validation (src/middleware/validateWebhook.ts)

Performs multi-layer validation of incoming webhooks:

  • Verifies HMAC signature or shared secret
  • Validates payload against Zod schema
  • Checks symbol whitelist and side values
  • Ensures required fields are present and correctly typed
Rate Limiting (src/middleware/rateLimiter.ts)

Protects against abuse using Redis-backed sliding window:

  • Limits requests per IP address
  • Configurable window size and request count
  • Falls back to in-memory storage when Redis unavailable
  • Returns 429 Too Many Requests when limit exceeded

Service Layer

Order Service (src/services/orderService.ts)

Orchestrates the complete order execution pipeline:

  • Receives validated trading signals
  • Delegates signal integrity checks to Supertrend validator
  • Calculates position size using the position sizer
  • Routes execution to appropriate DEX client
  • Handles execution results and error propagation
Supertrend Validator (src/services/supetrendValidator.ts)

Ensures signal integrity and prevents stale/executed signals:

  • Validates signal timing and sequence
  • Checks for duplicate signal processing
  • Verifies price levels against current market (optional)
  • Ensures SL/TP logic consistency
Position Sizer (src/services/positionSizer.ts)

Calculates appropriate position sizes based on risk parameters:

  • Implements fixed-fractional position sizing
  • Considers account equity, risk percentage, and stop loss distance
  • Applies leverage and exchange-specific constraints
  • Rounds to exchange-acceptable precision

DEX Client Layer

Hyperliquid Client (src/dex/hyperliquid.ts)

Handles communication with Hyperliquid exchange:

  • Manages API authentication and request signing

  • Handles WebSocket connections for real-time data (optional)

dYdX v4 Client (src/dex/dydx.ts)

Handles communication with dYdX v4 exchange:

  • Manages Ethereum wallet integration for authentication
  • Implements order placement via CompositeClient
  • Handles sub-account management and positioning

Utility Layer

Configuration (src/utils/config.ts)

Centralized environment configuration management:

  • Loads and validates environment variables
  • Provides typed access to configuration values
  • Supports different environments (development, production)
  • Includes sensible defaults where appropriate
Logging (src/utils/logger.ts)

Structured logging with Winston:

  • Configurable log levels and transports
  • JSON-formatted logs for production, readable for development

4. Data Flow

The system processes TradingView signals through a well-defined pipeline:

Signal Processing Pipeline

  1. Ingress: TradingView sends HTTPS POST to /api/webhook
  2. Middleware Processing:
    • Rate limiting middleware checks request frequency
    • Validation middleware authenticates and validates payload
  3. Route Handling: Webhook controller receives validated signal data
  4. Service Orchestration:
    • Order service validates signal integrity
    • Position sizer calculates trade size
    • DEX client prepares exchange-specific order
  5. Exchange Communication: Signed order transmitted to DEX API
  6. Response Handling: Exchange response processed and result returned
  7. Egress: HTTP response sent back to TradingView (success/error)

Data Structures

Key data structures flow through the system:

  • TradingView Payload: { secret, side, symbol, price, sl, tp, leverage }
  • Validated Signal: Typed object with all fields verified
  • Order Request: Exchange-specific parameters (size, price, type, etc.)
  • Exchange Response: Order ID, status, fees, and timestamps
  • Internal State: Configuration, rate limit counters, signal history

5. Module Details

Each module in the system has a clearly defined responsibility:

Configuration Module (utils/config.ts)

Responsible for managing application configuration:

  • Loads environment variables from .env or process environment
  • Provides type-safe accessors for all configuration values
  • Includes validation for critical settings (ports, secrets, API keys)
  • Supports environment-specific overrides
  • Exports singleton configuration instance

Logging Module (utils/logger.ts)

Provides centralized logging capabilities:

  • Winston logger with multiple transports (console, file)
  • Different formatting for development (pretty-print) vs production (JSON)
  • Configurable log levels per environment
  • Metadata enrichment for request tracing
  • Error logging with stack trace preservation

Webhook Validation Module (middleware/validateWebhook.ts)

Ensures only legitimate TradingView signals are processed:

  • Verifies shared secret or HMAC signature
  • Validates payload structure using Zod schema
  • Checks symbol against whitelist (BTC, ETH, etc.)
  • Validates side values (buy, sell, close)
  • Ensures numeric values are within reasonable bounds
  • Returns detailed validation errors for debugging

Rate Limiting Module (middleware/rateLimiter.ts)

Protects the service from abuse and overload:

  • Implements sliding window algorithm using Redis
  • Falls back to in-memory store when Redis unavailable
  • Tracks requests per IP address
  • Configurable limits (default: 10 requests/minute)
  • Returns HTTP 429 with retry-after header when limit exceeded

Order Service Module (services/orderService.ts)

Orchestrates the complete trading workflow:

  • Receives validated signal from webhook handler
  • Calls Supertrend validator to check signal integrity
  • Delegates position calculation to position sizer
  • Selects appropriate DEX client based on configuration
  • Formats exchange-specific order parameters
  • Handles execution results and error conversion
  • Logs all significant events for audit trail

Supertrend Validator Module (services/supetrendValidator.ts)

Validates the quality and integrity of trading signals:

  • Prevents processing of duplicate signals
  • Checks signal timing (stale signal detection)
  • Validates price relationships (entry < TP for longs, etc.)
  • Ensures stop loss is on correct side of entry
  • Can optionally verify signal against current market price
  • Maintains signal history for deduplication

Position Sizer Module (services/positionSizer.ts)

Calculates appropriate trade sizes based on risk management:

  • Implements fixed-fractional position sizing algorithm
  • Risk calculation: position size = (equity × risk%) / |entry - stop loss|
  • Applies leverage multiplier to calculated size
  • Respects exchange minimum order sizes and precision requirements
  • Returns size in base currency (e.g., BTC amount)
  • Logs risk parameters for audit purposes

DEX Clients (dex/)

Exchange-specific implementations for order execution:

Hyperliquid Client

  • Uses REST API with timestamp-based authentication
  • Implements POST /order endpoint for market and limit orders
  • Handles response parsing for order IDs and status
  • Error handling for insufficient funds, invalid prices, etc.
  • Optional WebSocket integration for real-time data

dYdX v4 Client

  • Leverages @dydxprotocol/v4-client-js library
  • Requires Ethereum wallet for API key signing
  • Uses CompositeClient.placeOrder for order submission
  • Supports both market and limit order types
  • Handles sub-account specification for positioning
  • Manages API credential lifecycle

6. Security Considerations

The bridge implements multiple layers of security to protect against unauthorized access and misuse:

Authentication and Authorization

  • Webhook Secret Validation: All incoming requests must contain a valid secret matching WEBHOOK_SECRET
  • Environment Isolation: Sensitive credentials (API keys, private keys) stored in environment variables
  • Principle of Least Privilege: DEX API keys limited to trading permissions only

Input Validation and Sanitization

  • Schema Validation: Zod schema validates all incoming webhook payloads
  • Type Safety: TypeScript ensures data integrity throughout processing
  • Range Checking: Numerical values validated against reasonable bounds
  • Symbol Whitelisting: Only pre-approved trading symbols accepted

Rate Limiting and Abuse Prevention

  • Per-IP Rate Limiting: Prevents brute force and flooding attacks
  • Sliding Window Algorithm: Provides smooth rate limiting behavior
  • Redis Backend: Distributed rate limiting for horizontal scaling
  • Fallback Mechanism: In-memory limiting ensures protection even when Redis unavailable

Secure Communications

  • HTTPS Enforcement: All external communications use TLS
  • API Key Protection: Exchange credentials never logged or exposed
  • Secret Management: Recommendation to use Docker secrets or vault in production
  • Network Segmentation: Suggestion to run behind reverse proxy with TLS termination

Error Handling and Information Disclosure

  • Generic Error Messages: Production errors return minimal information to users
  • Detailed Internal Logging: Full error details preserved in logs for debugging
  • Exception Handling: Global error handlers prevent stack trace leakage
  • Graceful Degradation: Service continues operating despite non-critical failures

7. Deployment Architecture

The bridge is designed for flexible deployment across various environments:

Containerized Deployment (Docker)

Recommended production deployment using Docker Compose:

docker-compose.yml contents: version: '3.8' services: app: build: . ports: - "3000:3000" environment: - NODE_ENV=production - PORT=3000 - WEBHOOK_SECRET=${WEBHOOK_SECRET} - PRIVATE_KEY=${PRIVATE_KEY} - DEFAULT_DEX=hyperliquid - REDIS_URL=redis://redis:6379 restart: unless-stopped redis: image: redis:7-alpine ports: - "6379:6379" restart: unless-stopped
  • Multi-stage Docker build for minimal production image
  • Separate Redis service for rate limiting persistence
  • Environment-specific configuration via .env file
  • Health checks and restart policies for resilience
  • Non-root user execution for enhanced security

Process Management

Production process management considerations:

  • Node.js cluster mode for multi-core utilization (optional)
  • PM2 or systemd for process supervision
  • Log rotation and retention policies
  • Monitoring and alerting for key metrics (request rate, error rate, latency)
  • Backup and recovery procedures for configuration

Scaling Considerations

Architectural elements that support horizontal scaling:

  • Stateless request processing enables load balancing
  • Externalized state (Redis) for rate limiting across instances
  • Configuration via environment variables supports multiple deployments
  • No affinity requirements - any instance can handle any request
  • Database-independent design facilitates easy replication

Development vs Production Differences

  • Development: Hot reloading, verbose logging, no rate limiting by default
  • Production: Optimized builds, minimal logging, strict rate limiting, error privacy
  • Configuration: Environment-specific .env files or secret management
  • Monitoring: Production includes metrics collection and health checks
